Pressure Sensor Array

CFDLabViewMATLABNI cDAQWiring
Timeline
3 months
Role
Lead Engineer
Overview

With assistance from my team, I created a 16 pressure sensor array used to create a pressure map of the front of our car's radiator. This was needed in order to create a realistic model of our radiator in CFD for our sidepod design.

I designed the setup from scratch, led a team of engineers, created the PCB design, selected the appropriate sensors, and set up the entire system before our planned testing day.

Images
Screenshot 1

Live dashboard

Screenshot 2

Latency metrics

Since our Formula SAE car never utilized sidepods before, we had to collect data in order to determine the porosity of our radiator, as that information was not readily available. This was crucial to verifying our CFD simulations in order to optimize the sidepod shroud. My goal was to use pressure sensors in a grid-pattern to measure the differential pressure at different points in front of the radiator, kind of how a pitot tube and static port work on a plane.
My teammate and I went through the development of a 3d printed grid, with 32 brass tubes, 16 pairs for 16 sensors. The sensors we used were 7391-BCE-S-005-000 differential pressure sensors, each with two inputs, one for ram air pressure and the other for static pressure. The static pressure tube was blocked on the end with holes drilled on the sides to act like static ports. This gave the pressure difference between the ambient pressure and the pressure received by the radiator based on the speed of the car, as ram-pressure increased with speed.
The biggest challenge was integrating the sensors with a brand new DAQ computer that we had never tested. I had to learn LabVIEW on-the-go from scratch using National Instruments' modules and controllers. I was able to get it running with only hours to spare as it was a last minute decision to switch, but it turned out to be a huge success.
We took CB6 to Colorado National Speedway, where we conducted straight-line speed tests, both with the wind rake in front and then behind the radiator. The wind-rake melted on our last test, but we were able to collect valuable data and ultimately determine the porosity of the radiator. We will now go on to finalize the sidepod design and begin manufacturing in the spring.

Key Challenges
  • Shifting from Arduino setup to NI cDAQ setup in a single day.
  • Tubes from sensors to inlets were not as bendable as hopes, so the array was offset a few inches from the front of the radiator
  • Harder for array to mount on the rear due to large tubes, so metal brackets were extended and can be flipped to the rear.
Procedures + Outcomes
  • Collected pressure data for three constant speeds by redlining up to 3rd gear.
  • Straight line, skidpad, S-turn, and Shalom courses driven.
  • Data was inputted into MATLAB to create pressure maps of front and rear faces of the radiator.

Gravity Anomaly Detection Satellite

MATLABASTROPHYSICSGROOPS
Timeline
Continuous
Role
Sole Author
Overview

Using previously obtained GRACE Satellite data, I'm developing a new design of gravity anomaly detection to predict and monitor climate change impacts. The design utilizes a single satellite with an atomic clock to precisely measure gravity changes based on time relativistic data, working off Einstein's theory of relativity. Current GRACE satellite missions consists of two satellites measuring distance changes due to gravity using laser-ranging and microwave-ranging technology, which is slightly outdated and satellites take time to accelerate, delaying the gravity measurement. Time change is nearly instantaenous and is far mroe accurate, allowing for high-degree data.

I am currently working on a MATLAB script to recreate GRACE satellite orbits to visualize how the frequency of atomic clocks change due to gravity data. This will allow for proof of concept while I write a paper on future designs and what technology will be required.

Technical Detail

Current GRACE satellite utilize distance montoring between two satellites to measure gravity changes across the globe. By using laser-ranging technology, the satellites can measure changes in distances down to a micrometer. However, the delay the satellites to move from each other and the cost manfuacturing two satellites begs the question if there is an alternative. I am currently working on a new design to implement a singular satellite GRACE missions using time dilation to measure gravity changes. By using atomic clocks, gravity changes can be measured instantaneously, reducing operational cost and manfuacturing and launch cost. This would also give far more details models of the Earth's gravitational field with the same overall goal of monnitoring and predicting climate change effects such as ice sheet mass shifts and sea level rise. Currently I am developing MATLAB simulatiosn to accurately showcase how atomic clock frequency would change based on previously collected GRACE satellite data. By doing this, I can get a more clear idea how effective the method is and how timing errors are affected using thiss method. Additionally, it provides visuals for those learning about such missions, as well as having a comparison to current models. This project is a continous project that I hope to finish before I complete my undergrad.

Key Features
  • Variable orbital element implentation, displaying orbital path with colormap to display frequency change across any orbit.
  • Specifies locations of increase frequency change of atomic clocks based on any GRACE data model.
  • Fully tunable for any dataset, color map, or choose between gravtational map or frequency map.

Timeline

Overview

A secure, atomic over-the-air firmware update system for resource-constrained ARM Cortex-M4 microcontrollers used in automotive ECUs. Designed to meet ISO 26262 functional safety standards with a bootloader footprint under 32KB.

Provides cryptographic signature verification, A/B partition rollback, and power-fail safety — all on hardware with no MMU.

Safety & Security
  • Ed25519 signature verification on every firmware image before any flash write begins — unsigned images are silently rejected.
  • Atomic A/B partition scheme: running firmware is never overwritten. A watchdog-guarded boot vote commits the new image only after successful startup.
  • Power-fail safe: CRC-verified write journal ensures partial writes are detected and rolled back on next boot.
  • All crypto uses mbedTLS with hardware RNG and side-channel-resistant scalar multiplication.

Overview

A high-throughput ML model serving layer designed to maximise GPU utilisation in production. The engine dynamically batches incoming inference requests, applies hardware-aware quantisation, and routes traffic across model versions for A/B experiments.

Open-sourced after internal deployment and adopted by several teams for serving transformer-based models at scale.

Key Features
  • Dynamic batching with configurable latency budget — collects requests for up to N ms before dispatching a fused GPU kernel call.
  • INT8 and FP16 quantisation with automatic calibration, reducing GPU memory footprint 2–4× with negligible accuracy loss.
  • A/B routing with configurable traffic splits and per-variant telemetry.
  • Triton Inference Server backend with a FastAPI facade for clean REST and gRPC interfaces.
